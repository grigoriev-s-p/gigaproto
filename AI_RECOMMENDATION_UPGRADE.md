# Что изменено

## 1) Рекомендации теперь генерирует AI-агент
Файл: `backend/recommendation_agent.py`

Старая версия строила рекомендации по алгоритмическим эвристикам.
Теперь рекомендации формирует отдельный AI-агент, который анализирует:
- `CURRENT_UI_PREVIEW` как главный источник истины,
- `CURRENT_UI_SCHEMA` как технический контекст,
- требования как продуктовый фон.

Он возвращает только структурированные и применимые рекомендации:
- `title`
- `scope`
- `priority`
- `description`
- `edit_prompt`

Это делает рекомендации заметно конкретнее и сразу пригодными для автоматического применения.

## 2) Подтверждение/отклонение рекомендаций тоже стало AI-driven
Файл: `backend/recommendation_edit_bridge.py`

Раньше мост между рекомендациями и правками работал в основном на регулярках и наборах фраз.
Теперь отдельный AI-агент интерпретирует ответы пользователя и решает:
- применить все рекомендации,
- применить только выбранные,
- применить рекомендации + пользовательские правки,
- проигнорировать рекомендации,
- выполнить только пользовательскую правку.

Это лучше работает на живых и нестандартных формулировках пользователя.

## 3) Агент правок усилен через AI-планировщик
Файл: `backend/ui_edit_agent.py`

Добавлен промежуточный AI-планировщик, который перед генерацией patch-а строит `EDIT_PLAN`:
- scope правки,
- целевые страницы,
- целевые секции,
- must_preserve,
- набор конкретных шагов.

Потом основной агент правок получает этот план вместе с текущим preview/schema.
Это особенно полезно для сложных задач:
- массовые изменения по всем страницам,
- удаление секций,
- перестановка блоков,
- сложные составные правки,
- mixed style + structural edits.

## 4) UI сообщений в чате стал понятнее
Файл: `frontend/src/app/App.tsx`

Текст рекомендаций теперь выводится в более читаемом формате:
- что за рекомендация,
- где именно,
- почему это важно,
- что конкретно сделать,
- приоритет.

## 5) Что проверено
- `python3 -m py_compile backend/*.py` — прошло.
- `node ./node_modules/typescript/bin/tsc -b` — прошло.

Полный runtime-поток с живыми ответами модели в контейнере не прогонялся, потому что он зависит от внешнего LLM API.
